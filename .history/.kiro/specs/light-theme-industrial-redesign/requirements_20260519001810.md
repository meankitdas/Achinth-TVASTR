# Requirements Document

## Introduction

This feature redesigns the Tvastr marketing site and customer portal from the current dark industrial aesthetic to a light industrial aesthetic, while introducing a coordinated animation system (GSAP for scroll-linked and hero choreography; Framer Motion for component-level transitions and gestures) and reconciling the new visual language with the existing centralized design-token system in `src/design/`. The redesign adopts component primitives and brand tokens scaffolded by `npx getdesign@latest add cohere`, mapped onto the existing `colors.js` token contract so the Python Tkinter Theme Editor and Tailwind config continue to function without breaking changes. The result is a light-themed surface that still reads as industrial (exposed grid, blueprint guides, monospace technical readouts, machined corners) and that introduces subtle, personal micro-interactions (cursor cues, magnetic CTAs, ambient idle motion, scroll-linked reveals) without compromising performance, accessibility, tier-gating semantics, or reduced-motion compliance.

## Glossary

- **Tvastr_Site**: The combined React + Vite application served from `src/`, including the public marketing pages and the authenticated `/portal` routes.
- **Light_Theme**: The new default color scheme that replaces the current dark scheme as the application's only theme upon completion of this feature.
- **Token_System**: The set of files under `src/design/` (`colors.js`, `typography.js`, `spacing.js`, `shadows.js`, `gradients.js`, `index.js`) plus the CSS variables in `src/index.css` and the import map in `tailwind.config.js`.
- **Cohere_Kit**: The set of component primitives, brand colors, and design tokens produced when running `npx getdesign@latest add cohere` in the workspace.
- **Brand_Color**: The primary accent color sourced from Cohere_Kit and surfaced as `colors.telemetry.primary` in `src/design/colors.js`.
- **Theme_Editor**: The Python Tkinter desktop application at `tools/theme_editor.py` that parses and rewrites hex values in `src/design/colors.js`.
- **GSAP_Layer**: The animation subsystem implemented using the GSAP library and its ScrollTrigger plugin, used for scroll-linked timelines, hero choreography, and section-spanning sequences.
- **Motion_Layer**: The animation subsystem implemented using the Framer Motion library, used for component-level enter/exit transitions, layout animations, gesture responses, and page transitions.
- **Industrial_Decor**: The set of light-theme visual motifs comprising fixed background grid lines, blueprint corner guides, monospace technical readouts, machined (clipped/notched) corners, and ruler-tick separators.
- **Forge_Core**: The existing Three.js icosahedron rendered in the hero via `src/three/ForgeCore.jsx` (and `FloatingGeometry.jsx`).
- **Tier_Badge**: The state badge rendered on portal product cards indicating one of `ACTIVE`, `INCLUDED`, or `LOCKED` based on the current license tier.
- **Locked_State**: The visual and behavioral state shown for features the current user's tier does not grant access to, rendered by `LockedProductCard`, `LockedFeatureBlock`, and `LockedScreen`.
- **Reduced_Motion**: The user preference detected by the `prefers-reduced-motion: reduce` media query.
- **Contrast_Ratio**: The WCAG 2.1 luminance contrast ratio between a foreground and background color.
- **Theme_Token**: A named entry inside the `colors` or `semantic` exports of `src/design/colors.js` (e.g., `colors.text.primary`, `semantic.brand`).
- **Animation_Frame_Budget**: The maximum permitted main-thread time consumed by animation work per frame on the target device class.

## Requirements

### Requirement 1: Adopt Cohere brand kit and reconcile with the existing token system

**User Story:** As a designer maintaining Tvastr's visual identity, I want the Cohere_Kit's brand assets and component primitives to drive the redesign while still flowing through `src/design/colors.js`, so that the Theme_Editor, Tailwind utilities, and CSS variables continue to resolve consistently.

#### Acceptance Criteria

1. THE Tvastr_Site SHALL execute `npx getdesign@latest add cohere` once during this feature's implementation and commit the resulting Cohere_Kit artifacts under a single, documented directory.
2. THE Tvastr_Site SHALL map the Cohere_Kit Brand_Color to `colors.telemetry.primary` in `src/design/colors.js`.
3. THE Tvastr_Site SHALL preserve the six existing top-level keys in the `colors` export of `src/design/colors.js` (`background`, `text`, `telemetry`, `process`, `signal`, `border`) and SHALL preserve the three keys of the `semantic` export (`brand`, `alert`, `danger`).
4. WHEN the Theme_Editor parses `src/design/colors.js` after the redesign, THE Theme_Editor SHALL successfully read every Theme_Token whose value is a 6-digit hex code without raising a parse error.
5. WHEN the Theme_Editor saves a 6-digit hex value to any Theme_Token, THE Tvastr_Site SHALL render that updated value through the corresponding Tailwind utility class on the next development build.
6. THE Tvastr_Site SHALL document, in the design directory, the mapping from each Cohere_Kit token to its Theme_Token destination.

### Requirement 2: Establish the Light_Theme color system

**User Story:** As a visitor, I want the site to present a coherent light industrial color palette, so that the brand reads as precise and modern in daylight viewing conditions.

#### Acceptance Criteria

1. THE Tvastr_Site SHALL set `colors.background.primary` to a hex value whose CIE relative luminance is greater than or equal to 0.85.
2. THE Tvastr_Site SHALL set `colors.text.primary` to a hex value that achieves a Contrast_Ratio of at least 7.0 against `colors.background.primary`.
3. THE Tvastr_Site SHALL set `colors.text.secondary` to a hex value that achieves a Contrast_Ratio of at least 4.5 against `colors.background.primary`.
4. THE Tvastr_Site SHALL set `colors.text.muted` to a hex value that achieves a Contrast_Ratio of at least 3.0 against `colors.background.primary`.
5. THE Tvastr_Site SHALL set `colors.telemetry.primary` to a hex value that achieves a Contrast_Ratio of at least 4.5 against `colors.background.primary` when used for body-size text or interactive labels.
6. THE Tvastr_Site SHALL set `colors.signal.warning` and `colors.signal.danger` to hex values that achieve a Contrast_Ratio of at least 3.0 against `colors.background.primary` when used as iconography or border accents.
7. THE Tvastr_Site SHALL define `colors.border.subtle`, `colors.border.default`, and `colors.border.strong` as `rgba()` values whose alpha is strictly increasing across the three keys.
8. THE Tvastr_Site SHALL update the CSS custom properties in `src/index.css` so that every `--*` variable resolves to the same value as its corresponding Theme_Token in `src/design/colors.js`.

### Requirement 3: Remove dark-theme defaults

**User Story:** As an engineer maintaining the codebase, I want the dark-theme styling removed so that there is exactly one source of truth for theme colors and no orphaned dark-mode classes.

#### Acceptance Criteria

1. THE Tvastr_Site SHALL render Light_Theme as the only theme served at runtime after the redesign is shipped.
2. THE Tvastr_Site SHALL NOT include any Tailwind `dark:` variant utility classes in committed JSX or CSS files after the redesign.
3. WHERE the operating system reports a `prefers-color-scheme: dark` preference, THE Tvastr_Site SHALL still render Light_Theme.
4. THE Tvastr_Site SHALL remove unused dark hex values from `src/design/colors.js` only by overwriting them with their Light_Theme replacements, preserving the six-category structure described in Requirement 1.

### Requirement 4: Install and isolate the animation libraries

**User Story:** As an engineer building the new motion system, I want GSAP and Framer Motion installed with clear roles, so that animation work is consistent and tree-shakable.

#### Acceptance Criteria

1. THE Tvastr_Site SHALL add `gsap` and `framer-motion` as production dependencies in `package.json`.
2. THE Tvastr_Site SHALL register the GSAP ScrollTrigger plugin exactly once at application startup.
3. THE GSAP_Layer SHALL be the only animation layer used for scroll-linked timelines that span more than one DOM section.
4. THE Motion_Layer SHALL be the only animation layer used for component mount/unmount transitions, layout animations, and pointer/tap gesture responses.
5. WHERE a single visual effect requires both a scroll trigger and a gesture response, THE Tvastr_Site SHALL drive the scroll portion with the GSAP_Layer and the gesture portion with the Motion_Layer, without animating the same CSS property from both layers simultaneously.
6. THE Tvastr_Site SHALL increase the gzipped production JS bundle by no more than 90 KB attributable to the addition of `gsap` and `framer-motion` combined.

### Requirement 5: Honor the Reduced_Motion preference

**User Story:** As a user with motion sensitivity, I want non-essential motion to be disabled when I have requested reduced motion, so that the site is comfortable to use.

#### Acceptance Criteria

1. WHEN Reduced_Motion is active, THE GSAP_Layer SHALL set the duration of every non-essential timeline to 0 and SHALL apply only the timeline's final state.
2. WHEN Reduced_Motion is active, THE Motion_Layer SHALL disable enter, exit, and idle animations and SHALL render components in their final visual state.
3. WHEN Reduced_Motion is active, THE Tvastr_Site SHALL still execute focus-visible and active-state transitions whose total duration is less than or equal to 150 milliseconds, because such transitions are essential affordances.
4. WHEN Reduced_Motion is active, THE Forge_Core SHALL render a single static frame and SHALL NOT advance its rotation animation.
5. WHEN Reduced_Motion is active, THE Tvastr_Site SHALL NOT render any continuous ambient particle motion.
6. WHEN the user toggles the Reduced_Motion preference at the operating-system level, THE Tvastr_Site SHALL apply the new preference within one navigation or page reload without requiring a manual page refresh once the preference change event fires.

### Requirement 6: Adopt the Industrial_Decor visual language in Light_Theme

**User Story:** As a visitor, I want the site to feel like an industrial blueprint even in light mode, so that Tvastr's positioning as an industrial AI company is unmistakable.

#### Acceptance Criteria

1. THE Tvastr_Site SHALL render a fixed background grid layer behind primary content surfaces using lines drawn from `colors.border.subtle`.
2. THE Tvastr_Site SHALL render blueprint corner guides at the four corners of the viewport on routes whose path matches `/`, `/technology`, `/about`, or `/research`, with stroke color sourced from `colors.border.default`.
3. THE Tvastr_Site SHALL render section eyebrows, version strings, checksums, coordinate readouts, and other technical metadata in a monospace typeface.
4. THE Tvastr_Site SHALL apply a machined-corner clip (notched or 45-degree chamfered) to at least one corner of every primary card surface (`ProductCard`, `LockedProductCard`, `ProductDownloadCard`, `RollbackVersionCard`, `UpgradeBanner`).
5. THE Tvastr_Site SHALL render section dividers as ruler-tick lines composed of evenly spaced short marks against `colors.border.default`.
6. THE Industrial_Decor layer SHALL be rendered behind interactive content with a stacking context that does not capture pointer events.

### Requirement 7: Adapt the Forge_Core for Light_Theme

**User Story:** As a visitor on the home hero, I want the existing Three.js forge core to remain visually present and on-brand against a light background, so that the hero retains its signature.

#### Acceptance Criteria

1. THE Forge_Core SHALL render against the Light_Theme background without producing a Contrast_Ratio of less than 1.4 between its silhouette edges and `colors.background.primary` at any frame.
2. THE Forge_Core SHALL source its primary emissive accent color from `semantic.brand`.
3. WHEN the viewport width is less than 768 pixels, THE Forge_Core SHALL reduce its rendered pixel ratio to a value not greater than 1.5.
4. THE Forge_Core SHALL maintain a sustained frame rate of at least 50 frames per second on a desktop reference device with a 60 Hz display when no other tab-blocking work is active.
5. WHEN the hero section's bounding rectangle is fully outside the viewport, THE Forge_Core SHALL pause its render loop.

### Requirement 8: Choreograph the Hero on entry and on scroll

**User Story:** As a visitor landing on the home page, I want the hero to assemble with deliberate, smooth motion and respond to my scroll, so that the first impression communicates precision.

#### Acceptance Criteria

1. WHEN the home page mounts, THE GSAP_Layer SHALL run an entry timeline that reveals the wordmark, tagline, primary CTA, and Forge_Core in a defined order with a total duration between 800 and 1600 milliseconds.
2. WHILE the user scrolls within the hero section, THE GSAP_Layer SHALL drive a scroll-linked parallax on the Forge_Core whose translation magnitude does not exceed 120 pixels at the hero's exit boundary.
3. WHEN the hero entry timeline is active, THE Tvastr_Site SHALL keep all interactive elements (CTA button, navbar links) responsive to pointer and keyboard input without blocking.
4. IF the hero entry timeline is interrupted by route navigation, THEN THE GSAP_Layer SHALL kill the timeline and release its references before the next page mounts.

### Requirement 9: Provide scroll-linked reveals for every section

**User Story:** As a visitor scrolling the home and technology pages, I want sections to reveal smoothly as they enter the viewport, so that the page feels alive without being distracting.

#### Acceptance Criteria

1. WHEN any section in `src/components/sections/` enters the viewport, THE GSAP_Layer SHALL fade and translate its primary heading and subheading into their final position over a duration between 400 and 900 milliseconds.
2. THE GSAP_Layer SHALL trigger each section's reveal exactly once per page lifetime.
3. WHEN a section's reveal completes, THE GSAP_Layer SHALL clear its inline transform and opacity styles from the section's primary heading and subheading.
4. WHERE a section contains a `FeatureGrid` primitive, THE GSAP_Layer SHALL stagger the reveal of its grid items with an inter-item delay between 40 and 120 milliseconds.

### Requirement 10: Provide page transitions via the Motion_Layer

**User Story:** As a visitor navigating between routes, I want a brief, consistent transition between pages, so that navigation feels intentional.

#### Acceptance Criteria

1. WHEN the route changes between any two routes registered in `App.jsx`, THE Motion_Layer SHALL animate the outgoing page out and the incoming page in with a combined duration not greater than 500 milliseconds.
2. THE Motion_Layer SHALL preserve scroll position behavior consistent with React Router's existing behavior across the transition.
3. IF a page transition is in progress when a new route change is requested, THEN THE Motion_Layer SHALL cancel the in-progress transition and start the new transition from the current visual state.

### Requirement 11: Provide subtle micro-interactions

**User Story:** As a visitor, I want small responsive cues from the interface, so that the site feels personal and crafted without being noisy.

#### Acceptance Criteria

1. THE Motion_Layer SHALL apply a magnetic hover effect to every primary CTA button such that the button's translation toward the cursor does not exceed 8 pixels in either axis.
2. WHEN the pointer hovers over an element annotated as magnetic, THE Motion_Layer SHALL spring the element back to its origin within 250 milliseconds after the pointer leaves.
3. THE Tvastr_Site SHALL render a subtle cursor follower whose maximum radius is 24 pixels and whose opacity does not exceed 0.35.
4. WHEN the pointer is idle for at least 2 seconds, THE Motion_Layer SHALL fade the cursor follower to 0 opacity over a duration between 200 and 600 milliseconds.
5. WHILE the user has not interacted with the page for at least 8 seconds, THE GSAP_Layer SHALL drive an ambient idle pulse on the Forge_Core whose scale variation is between 0.98 and 1.02.
6. WHERE a device reports no fine pointer (`(pointer: coarse)`), THE Tvastr_Site SHALL NOT render the cursor follower and SHALL NOT apply the magnetic hover effect.

### Requirement 12: Style and animate the Navbar

**User Story:** As a visitor, I want the top navigation to feel anchored and responsive as I scroll, so that I always know where I am.

#### Acceptance Criteria

1. WHEN the document scroll offset is less than 16 pixels, THE Navbar SHALL render with a transparent background.
2. WHEN the document scroll offset is greater than or equal to 16 pixels, THE Navbar SHALL render with a background color sourced from `colors.background.primary` at an alpha between 0.85 and 0.95 and a 1-pixel bottom border sourced from `colors.border.subtle`.
3. WHEN a navigation link receives keyboard focus, THE Tvastr_Site SHALL render a visible focus ring whose Contrast_Ratio against `colors.background.primary` is at least 3.0.
4. THE Navbar SHALL render the active route's link with a foreground color sourced from `colors.telemetry.primary`.

### Requirement 13: Apply Light_Theme and motion to home-page sections

**User Story:** As a visitor on the home page, I want every section to share a consistent light industrial treatment with smooth reveals, so that the page reads as one continuous document.

#### Acceptance Criteria

1. THE HeroSection, AboutSection, EcosystemSection, DeploymentSection, ProductSlider, IndustryProblemSection, InspectionVisibilitySection, PlatformOverviewSection, and ContactSection SHALL each render against `colors.background.primary` with no remaining hard-coded dark hex values in their JSX or local stylesheets.
2. THE EcosystemSection SHALL render its flow diagram strokes using `colors.process.primary` and node fills using `colors.background.primary`.
3. THE DeploymentSection SHALL render its on-premise architecture strokes using `colors.telemetry.primary` and SHALL apply Industrial_Decor ruler-tick separators between architectural zones.
4. THE ProductSlider SHALL animate horizontal card transitions through the Motion_Layer with a per-transition duration not greater than 450 milliseconds.
5. WHEN a ProductCard's animated visual (defect grid or data flow) is on screen, THE GSAP_Layer SHALL drive that visual's animation; otherwise, THE Tvastr_Site SHALL pause that visual's animation.
6. THE ContactSection SHALL render the founder's email and LinkedIn link with focus-visible underlines using `colors.telemetry.primary`.

### Requirement 14: Apply Light_Theme and motion to system detail pages

**User Story:** As a visitor reading a system detail page, I want technical content to feel like a precise blueprint document, so that the engineering substance comes through.

#### Acceptance Criteria

1. THE RejectionAnalysisSystem and PlantIntelligence pages SHALL render against `colors.background.primary` and SHALL apply Industrial_Decor blueprint corner guides on viewports wider than 1024 pixels.
2. WHEN a SystemImageBlock enters the viewport, THE GSAP_Layer SHALL reveal it with a duration between 400 and 900 milliseconds.
3. THE SystemWorkflow component SHALL render its connector lines using `colors.border.strong` and its node labels using `colors.text.primary`.
4. THE SystemImpactGrid component SHALL render impact metric tiles with a machined-corner clip on at least one corner of each tile.

### Requirement 15: Apply Light_Theme to the portal and preserve tier-gating UX

**User Story:** As a customer signed in to the portal, I want the dashboard, downloads page, and locked screens to read clearly in Light_Theme, so that my license state and download options are unambiguous.

#### Acceptance Criteria

1. THE PortalLogin, PortalDashboard, and PortalDownloads pages SHALL render against `colors.background.primary`.
2. THE Tier_Badge for an `ACTIVE` state SHALL render with a foreground sourced from `colors.signal.warning` and a border sourced from `colors.signal.warning` at alpha less than or equal to 0.5, regardless of which tier produced the active state.
3. THE Tier_Badge for an `INCLUDED` state SHALL render with a foreground sourced from `colors.text.muted` and SHALL display the literal label text `INCLUDED`.
4. THE Tier_Badge for a `LOCKED` state SHALL render with a foreground sourced from `colors.text.muted`, an icon sourced from the existing lock iconography, and an upgrade CTA whose accent is sourced from `colors.signal.warning`.
5. WHEN the user's license tier is `ras_core`, THE PortalDashboard SHALL render a `LOCKED` Tier_Badge on the RAS Enterprise card and on the Plant Intelligence card.
6. WHEN the user's license tier is `ras_enterprise`, THE PortalDashboard SHALL render an `ACTIVE` Tier_Badge on the RAS Enterprise card and a `LOCKED` Tier_Badge on the Plant Intelligence card.
7. WHEN the user's license tier is `full_stack`, THE PortalDashboard SHALL render an `ACTIVE` Tier_Badge on the Plant Intelligence card and SHALL render an `INCLUDED` Tier_Badge on the RAS Enterprise card.
8. WHERE the user's license tier is `full_stack`, THE PortalDashboard SHALL NOT render the UpgradeBanner.
9. WHEN the user navigates to a route guarded by a tier the user does not hold, THE LockedScreen SHALL render the lock icon, the feature title, the upgrade message, and the upgrade CTA, sourcing all colors from Theme_Tokens.

### Requirement 16: Theme-Token resolution invariants

**User Story:** As an engineer, I want guarantees that every Theme_Token referenced by the app resolves to a defined value, so that we never ship with a missing or undefined color.

#### Acceptance Criteria

1. FOR ALL Tailwind utility classes that reference a Theme_Token (`bg-*`, `text-*`, `border-*`, `fill-*`, `stroke-*`) used anywhere in `src/`, THE Tvastr_Site SHALL resolve the class to a non-empty CSS color string at build time.
2. FOR ALL CSS custom properties declared in `src/index.css` whose name begins with `--bg-`, `--text-`, `--telemetry-`, `--process-`, `--signal-`, or `--border-`, THE Tvastr_Site SHALL define a value that is either a 6-digit hex code or a valid `rgba()` expression.
3. THE Tvastr_Site SHALL produce no runtime references to undefined Theme_Tokens; if a referenced Theme_Token is missing, the build SHALL fail.
4. FOR ALL Theme_Tokens whose category is `background`, `text`, `telemetry`, `process`, or `signal`, THE Tvastr_Site SHALL store the value as a 6-digit hex code matching the regular expression `^#[0-9a-f]{6}$`.

### Requirement 17: Animations never block input

**User Story:** As a user, I want to be able to click and type at all times, so that motion never gets in my way.

#### Acceptance Criteria

1. WHILE any GSAP_Layer or Motion_Layer animation is running, THE Tvastr_Site SHALL keep all CTA buttons, navigation links, and form inputs reachable by pointer and keyboard.
2. WHILE any animation is running, THE Tvastr_Site SHALL NOT apply `pointer-events: none` to any element that contains a focusable descendant.
3. WHEN the user clicks a CTA whose magnetic hover animation is in progress, THE Tvastr_Site SHALL fire the click handler within the same event loop tick as a non-animated click would.
4. THE Tvastr_Site SHALL keep main-thread Animation_Frame_Budget consumption from animation work at or below 8 milliseconds per frame on a desktop reference device.

### Requirement 18: Responsive behavior across device classes

**User Story:** As a user on mobile, tablet, or desktop, I want the redesigned site and its motion to feel appropriate for my device, so that nothing feels broken or excessive.

#### Acceptance Criteria

1. WHEN the viewport width is less than 640 pixels, THE Industrial_Decor blueprint corner guides SHALL render at no more than 50% of their desktop stroke length.
2. WHEN the viewport width is less than 640 pixels, THE Tvastr_Site SHALL NOT render the cursor follower or the magnetic hover effect.
3. WHEN the viewport width is between 640 and 1024 pixels, THE GSAP_Layer SHALL reduce all section reveal durations by between 20% and 40% of their desktop values.
4. WHEN the viewport width is greater than or equal to 1024 pixels, THE Tvastr_Site SHALL render the full Industrial_Decor set as defined in Requirement 6.
5. WHEN the device reports `(hover: none)`, THE Tvastr_Site SHALL apply tap-state animations through the Motion_Layer instead of hover-state animations.

### Requirement 19: Performance budget for the redesign

**User Story:** As a product owner, I want the redesign to stay within a defined performance budget, so that we do not regress page-load metrics or runtime smoothness.

#### Acceptance Criteria

1. THE Tvastr_Site SHALL produce a production build whose total gzipped JS bundle increases by no more than 120 KB compared to the build immediately preceding this feature.
2. THE Tvastr_Site SHALL produce a production build whose total gzipped CSS bundle increases by no more than 30 KB compared to the build immediately preceding this feature.
3. WHEN the home page is loaded on a desktop reference device, THE Tvastr_Site SHALL maintain a Largest Contentful Paint not greater than 2.5 seconds and a Cumulative Layout Shift not greater than 0.1, measured against the production build.
4. WHEN any scroll-linked GSAP_Layer timeline is active, THE Tvastr_Site SHALL maintain a sustained frame rate of at least 55 frames per second on a desktop reference device.

### Requirement 20: Idempotent and confluent theme application

**User Story:** As an engineer running theme-related operations, I want theme application to be predictable, so that repeated runs of the same operation produce the same result.

#### Acceptance Criteria

1. WHEN the Theme_Editor saves the same color map twice in a row, THE Tvastr_Site SHALL produce identical bytes for `src/design/colors.js` after each save (idempotence).
2. WHEN the Theme_Editor exports a JSON theme, then imports the same JSON, then saves, THE Tvastr_Site SHALL produce a `src/design/colors.js` whose Theme_Token values equal the exported JSON's values for every key (round-trip property).
3. WHEN two non-overlapping Theme_Token edits are applied in either order, THE Tvastr_Site SHALL produce the same final `src/design/colors.js` content regardless of order (confluence).

### Requirement 21: Error handling for animation and theme resolution

**User Story:** As a user, I want the site to keep working even if the animation libraries or theme tokens fail to load, so that I never see a broken page.

#### Acceptance Criteria

1. IF the GSAP library fails to load, THEN THE Tvastr_Site SHALL render every section in its final visual state without scroll-linked animation and SHALL log a single warning to the console.
2. IF the Framer Motion library fails to load, THEN THE Tvastr_Site SHALL render every component in its final visual state without enter or exit transitions and SHALL log a single warning to the console.
3. IF a referenced Theme_Token resolves to an empty or undefined value at runtime, THEN THE Tvastr_Site SHALL fall back to `colors.text.primary` for foreground references and `colors.background.primary` for background references, and SHALL log a single warning to the console naming the missing token.
4. IF the Forge_Core's WebGL context is lost, THEN THE Forge_Core SHALL render a static SVG fallback whose accent color is sourced from `semantic.brand`.
