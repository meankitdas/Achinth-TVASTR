# Artifact Contracts

> **Purpose:** Formal contract defining all runtime artifacts produced by TvastrRAS system. Every output file, JSON schema, and telemetry stream is a stable, versioned interface used by downstream systems (ERP, MES, QA, analytics, dashboards). No implicit formats. No undocumented outputs.

> **Version:** 2.0  
> **Last Updated:** 2026-05-16

---

## Artifact Classification

All artifacts are categorized by lifecycle and consumption type:

| Category | Definition | Retention | Access |
|---------|------------|-----------|--------|
| **Core Output** | Primary inspection result artifacts required for system operation and regulatory traceability | 7+ years | Read-only by consumers |
| **Telemetry** | Raw diagnostic logs for monitoring, calibration, and failure analysis | 90 days | Read-only |
| **Batch Artifact** | Input/output files for batch processing workflows | 30 days | Read/write |
| **Metadata** | Supporting data for artifact interpretation and validation | Indefinite | Read-only |
| **Runtime Output** | File system structures generated by runtime | Indefinite | Read-only |

### 2. `annotated_surface.jpg` — Inspected Image Output
**Location**: `runtime/outputs/{casting_id}/{casting_id}_annotated.jpg`

**Purpose**: Human-readable visualization of detected defects on casting surface.

**Properties**:
- Resolution: Same as input (after resize to 960px max side)
- Overlay: Green bounding boxes (confidence > 0.7), yellow (0.5-0.7), red (<0.5)
- Annotations: Each box labeled with defect type + confidence
- Heatmap overlay: Semi-transparent (alpha=0.4) jet colormap showing anomaly score distribution
- File format: JPEG, quality=90
- Metadata embedded: EXIF tags for casting_id, decision, timestamp, and version="2.0"

---

### 3. `heatmap.jpg` — Anomaly Probability Overlay
**Location**: `runtime/outputs/{casting_id}/{casting_id}_heatmap.jpg`

**Purpose**: Pure anomaly heatmap (no input image).

**Properties**:
- Color scheme: Jet colormap (blue=low, red=high)
- Dimensions: 960×960 pixels
- Pixel values: Normalized [0,1] from `final_anomaly` scores
- No bounding boxes or labels
- Used by QA engineers for pattern recognition
- Filename includes version: `heatmap_v2.0.jpg` (embedded as metadata)

---

### 4. `report.pdf` — Human-Readable Inspection Report
**Location**: `runtime/outputs/{casting_id}/{casting_id}_report.pdf`

**Purpose**: Print/audit-ready report with full context.

**Structure**:
- Page 1: Casting ID, decision, timestamp, heat number — clearly marked “TvastrRAS v2.0”
- Page 2: Annotated image + heatmap side-by-side — labeled “Phase-K Energy-Based Reasoning”
- Page 3: Defect table: Type, Confidence, BBox, Zone, Severity
- Page 4: Root Cause Analysis with recommended actions — references energy-based classification
- Page 5: Technical summary: Model versions, processing time, pipeline_health
- Footer: System version v2.0, run ID, digital signature (SHA-256 hash of `report.json`)
- Watermark: "PHASE-K ENERGY-BASED CLASSIFICATION - DO NOT CONFUSE WITH PREVIOUS VERSIONS"

## Runtime Output Structures

### 1. Runtime Directory Hierarchy
**Structure**:
```
runtime/
├── batch_input/
│   ├── incoming/          # Batch input images
│   └── processed/         # Archived images after processing
├── outputs/
│   ├── reports/           # PDFs
│   └── exports/           # CSV, JSON
├── logs/
│   ├── ras.log            # Main app
│   ├── signal_traces/     # YYYY-MM-DD.jsonl
│   └── reconciliation.log # Heat resolution audit
├── process_logs/          # CSV fallback if SQL down
├── fingerprint_index/
│   ├── fingerprints.db    # SQLite index
│   └── vectors/           # Serialized vectors
├── calibration/
│   ├── weights_history.json
│   └── confidence_maps.pkl
└── updates/
    └── pending/
```

**Guarantees**:
- All directories created automatically on startup
- No writes outside these paths
- Files are never overwritten (new version appended)
- `outputs/` and `logs/` are the only directories consumers may read
- `classification_source` in all outputs = `"energy-based"`

## Critical Invariants (Must Always Hold)

1. **Version Consistency**: All artifacts are versioned. No undocumented fields.
2. **No Side Effects**: Artifacts are outputs only — system never reads its own artifacts during execution.
3. **Atomic Output**: Each artifact is written completely or not at all (no partial writes).
4. **Read-Only by Upstream**: ERP, UI, and analytics tools must not modify artifacts.
5. **Explicit Schema**: Every JSON file has an exact schema defined herein — no implicit fields.
6. **Energy-Based Tagging**: All outputs must reflect Phase-K energy-based reasoning. No references to 4-gate or YOLO classification.

## Error Cases

If an artifact violates this contract, the system must:
- Log: `[ARTIFACT_ERROR] Casting ID=CAST-XXX: Missing field 'decision' in report.json`
- Set `pipeline_health = "FAILED"`
- Send alert to `runtime/logs/ras.log`
- Do not proceed with export or ERP upload
- Preserve the file — never delete — for forensic analysis
